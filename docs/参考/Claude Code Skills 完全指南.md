## 1. 什么是 Skill？一句话理解

```
把 Claude 想象成一个全能助手。

普通对话 = 你每件事都要手把手教它，它凭记忆和推理来做
Skill    = 给它装了一个"自动化流程包"，遇到特定任务时，
          它知道标准步骤，自动按最佳实践执行，不用你一步步指挥
```

### 打个比方

| 场景 | 普通对话 | 使用 Skill |
|------|---------|-----------|
| **写代码** | 你："帮我写个函数"<br>Claude 写了<br>你："现在帮我测试一下"<br>Claude 写了测试<br>你："跑一下看看"<br>……一步步来 | 你："帮我实现这个功能"<br>Claude 自动：规划→写代码→写测试→跑通→报告结果 |
| **代码审查** | 你："帮我看看这代码有问题吗"<br>Claude 大概说几个问题 | 你："review 一下"<br>Claude 自动：逐文件审查→按严重程度分类→给出修复建议→可自动修复 |
| **部署验证** | 你："部署好了，帮我看看"<br>需要你手动描述状态 | 你："verify"<br>Claude 自动：跑应用→截图→对比预期→报告差异 |

## 2. Skill vs 普通对话：本质区别

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  普通对话模式                    Skill 模式                  │
│  ────────────                   ──────────                  │
│                                                             │
│  你 → 提问                    你 → /命令                    │
│  Claude → 回答                Claude → 加载专业流程         │
│  你 → 追问                           ↓                    │
│  Claude → 再回答              自动执行多步骤                 │
│  你 → 纠正                           ↓                    │
│  Claude → 修改                一站式完成                     │
│  ...循环...                   你 → 确认结果                 │
│                                                             │
│  适合：探索、讨论、简单修改    适合：标准化任务、复杂流程     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 核心区别

| 维度 | 普通对话 | Skill |
|------|---------|-------|
| **执行方式** | 人驱动，一步步 | 流程驱动，自动化 |
| **步骤** | 每次临时想下一步 | 有预设的最佳实践流程 |
| **一致性** | 看心情，可能遗漏步骤 | 每次都按同样标准执行 |
| **工具使用** | 手动指定 | 自动调用合适的工具链 |
| **适用场景** | 开放式讨论 | 标准化任务 |

## 3. 怎么使用 Skill？

### 方式一：斜杠命令（最常用）

在对话框中直接输入：

```
/code-review         代码审查
/run                 启动并运行应用
/review              审查 Pull Request
/simplify            代码简化优化
/verify              验证代码改动是否生效
/loop 5m /xxx        每5分钟自动执行某个命令
/init                初始化项目文档
```

### 方式二：用自然语言描述（Claude 自动识别）

你不用打斜杠，直接说话，Claude 会判断该用哪个 Skill：

```
你说："帮我审查一下刚才的改动"           → Claude 自动调用 code-review
你说："跑起来看看效果"                   → Claude 自动调用 run
你说："这代码能不能简化一下"             → Claude 自动调用 simplify
你说："帮我确认这个 bug 修好了没"       → Claude 自动调用 verify
你说："给我写一份这个技术方案的深度调研"  → Claude 自动调用 deep-research
```

### 方式三：我（Claude）主动建议

当我判断当前任务适合用某个 Skill 时，会主动提醒你：

```
"这个改动涉及多个文件，要不要我用 /code-review 帮你全面审查一下？"
```

## 5. 量化交易中你真正会用到的 Skill

根据你的场景（零基础、搭量化系统、写策略、跑数据），按使用频率排序：

### 🟢 每天都可能用到的

| Skill | 你用它做什么 | 频率 |
|-------|-------------|:----:|
| **code-review** | 写完策略代码后，让 Claude 检查有没有 bug、有没有逻辑错误 | 每次写新代码后 |
| **run** | 跑 `python run.py` 看信号输出、跑回测看结果 | 每次改完代码 |

### 🟡 阶段性用到的

| Skill | 你用它做什么 | 频率 |
|-------|-------------|:----:|
| **deep-research** | 研究新策略思路、调研数据源、对比技术方案 | 研究阶段 |

| **verify** | 改了一个 bug 后，确认真的修好了 | 修 bug 时 |

### 🔵 偶尔用到的

| Skill | 你用它做什么 | 频率 |
|-------|-------------|:----:|

| **update-config** | 配置自动权限、环境变量 | 一次性 |

| **claude-api** | 将来如果你想用 Claude API 做智能选股/新闻分析 | 进阶阶段 |
| **fewer-permission-prompts** | 减少烦人的权限弹窗 | 一次性配置 |

## 6. 每个 Skill 的详细用法

### 6.1 `/code-review` —— 代码审查

**它做什么：**
检查你刚改的代码，找出：
- 🐛 Bug（逻辑错误、边界条件遗漏）
- 🧹 冗余代码（可以简化或复用的地方）
- ⚡ 性能问题
- 📐 不符合项目风格的写法

**你什么时候用：**
```
- 写完一个新策略（如 momentum_breakout.py）后
- 改了数据下载逻辑后
- 改了风控过滤规则后
- 任何"不确定写得对不对"的时候
```

**怎么用：**
```
/code-review              # 审查所有未提交的改动
/code-review --fix        # 审查 + 自动修复发现的问题
/code-review --comment    # 把审查结果写成注释
```

**对你的量化项目的实际例子：**

```
你写完了一个新的动量突破策略 → /code-review
Claude 会检查：
  ✅ 突破判断逻辑对不对（有没有未来函数？）
  ✅ 边界条件（数据不够20天时怎么办？）
  ✅ 停牌处理是否正确
  ✅ 信号强度的计算公式是否合理
  ✅ 有没有变量名写错、引用错误
  ✅ 和现有策略基类的接口是否一致
```

---

### 6.2 `/run` —— 启动运行

**它做什么：**
自动运行你的项目，让你看到实际效果。对于你的量化项目：
- 运行 `python run.py` 并展示信号输出
- 运行数据下载脚本
- 运行回测

**你什么时候用：**
```
- 每次改完策略代码，想看实际信号
- 想确认数据下载是否正常
- 想看策略在某个日期会输出什么信号
```

**怎么用：**
```
/run                       # Claude 自动判断怎么启动你的项目
"帮我跑一下看看信号"        # 自然语言也行
```

**对你的量化项目的实际例子：**

```
你说："帮我跑一下 run.py，看看今天有哪些买入信号"

Claude 会：
  1. 激活虚拟环境
  2. 执行 python run.py
  3. 把输出结果展示给你
  4. 如果报错，自动分析原因
```

---

### 6.3 `/deep-research` —— 深度调研

**它做什么：**
针对一个具体问题，自动做多源搜索 → 交叉验证 → 输出一份有引用的调研报告。

**你什么时候用：**
```
- "A股有哪些可行权证策略？"
- "2025年动量因子在A股是否还有效？最新研究怎么说？"
- "vnpy vs backtrader vs zipline，哪个适合我的场景？"
- "A股T+0ETF有哪些？能做日内策略吗？"
```

**怎么用：**
```
/deep-research A股量化选股中，哪些因子在2024-2025年仍然有效？
```

**对你的量化项目的实际例子：**

```
你说："帮我调研一下A股的小市值因子，最近两年还有效吗"

Claude 会：
  1. 搜索学术论文、量化社区讨论、券商研报
  2. 收集关于小市值因子在A股表现的数据
  3. 交叉验证不同来源的结论
  4. 输出一份有引用来源的报告：
     - 小市值因子的历史表现
     - 2024-2025年是否衰减
     - 与其他因子组合的效果
     - 在A股实施的注意事项（壳价值、退市风险等）
```

---

### 6.4 `/simplify` —— 代码简化

**它做什么：**
只做代码质量优化——删冗余、提复用、改结构、加注释。**不找 bug**（那是 code-review 的工作）。

**你什么时候用：**
```
- 代码写了一段时间，越来越臃肿
- 发现多个策略里有重复的逻辑
- 想把数据下载逻辑重构一下
```

**怎么用：**
```
/simplify                  # 简化所有改动的代码
```

**对你的量化项目的实际例子：**

```
你的项目中：
  - run.py 和 engine/runner.py 都有获取股票数据的逻辑（重复了）
  - calculate_position() 的参数判断可以提取成公共函数
  - 三个策略都各自写了均线计算（应该抽到基类）

/simplify 会自动识别这些并帮你合并整理
```

---

### 6.5 `/verify` —— 验证改动

**它做什么：**
你改了一个 bug 或加了一个功能后，Claude 会实际运行应用，观察行为，确认改动确实生效了。

**和 `/run` 的区别：**
- `/run`：只是把应用跑起来给你看
- `/verify`：跑起来 + **拿着预期的行为去验证** + 告诉你"通过"或"不通过"

**你什么时候用：**
```
- 修了一个数据下载的 bug 后 → 确认真的能下载到数据了
- 加了ST过滤规则后 → 确认ST股票确实被排除了
- 改了止损计算逻辑后 → 确认止损价算对了
```

**怎么用：**
```
/verify                    # 验证最新的改动
```

**对你的量化项目的实际例子：**

```
你改了风险过滤器，让它排除科创板股票

你说："帮我验证一下"

Claude 会：
  1. 运行 python run.py
  2. 检查输出中是否还有 688xxx 的股票
  3. 如果有 → 告诉你"验证失败，还有科创板股票"
  4. 如果没有 → 告诉你"验证通过，科创板已全部排除"
```

---

### 6.6 `/security-review` —— 安全审查

**它做什么：**
检查代码的安全漏洞——对量化项目特别重要的是：
- 是否有硬编码的 API Key/Token
- 网络请求是否安全
- 数据库操作是否有注入风险
- 文件操作是否安全

**你什么时候用：**
```
- 实盘上线前（很重要！）
- 接入了券商 API 后
- 添加了网络请求功能后
```

**怎么用：**
```
/security-review
```

**对你的量化项目的实际例子：**

```
实盘前执行 /security-review，Claude 会检查：
  ✅ 券商 API Key 是否硬编码在代码里（应该放环境变量）
  ✅ 数据库操作有没有 SQL 注入风险
  ✅ 通知 webhook URL 有没有暴露
  ✅ 日志有没有意外记录敏感信息
```

---

### 6.7 `/loop` —— 定时循环

**它做什么：**
每隔一段时间自动执行一个命令。对量化项目最直观的用法：**每个交易日收盘后自动跑信号**。

**你什么时候用：**
```
- 每5分钟检查一次数据下载进度
- 每个交易日15:05自动运行策略
- 每30分钟检查实盘状态
```

**怎么用：**
```
/loop 5m /run               # 每5分钟跑一次
/loop 1d python run.py      # 每天跑一次（不精确到15:05）
```

**对你的量化项目的实际例子：**

```
# 收盘后自动生成信号（需要先配置好定时调度）
/loop 1d "收盘了，帮我跑一下 python run.py 看看信号"

# 实际上更推荐用 macOS 的 launchd 做定时，
# /loop 更适合开发调试阶段
```

### 6.8 `/init` —— 项目初始化

**你什么时候用：**
```
- 项目结构基本稳定后（现在就可以用！）
```

**怎么用：**
```
/init
```

## 7. 组合使用：一条龙工作流

### 场景一：开发一个新策略的完整流程

```
1. 你说：
   "帮我研究一下A股动量因子怎么做，写一个新策略"

2. Claude 可能建议先用 /deep-research：
   → 调研动量因子的定义、参数、A股实证
   → 输出调研报告

3. 确认方案后，Claude 写代码：
   → strategies/momentum_breakout.py
   → 在 engine/runner.py 中注册

4. 代码写完后：
   /code-review         → 检查 bug 和逻辑错误
   /run                 → 跑起来看信号
   /verify              → 确认信号符合预期

5. 满意后：
   /simplify            → 整理代码、消除重复
```

### 场景二：日常改代码 + 验证

```
1. 你改了策略参数（MA10/30 → MA5/20）
2. /code-review        → 快速检查有没有写错
3. /run                → 看新的信号输出
4. 对比新旧参数哪个信号更多/更少
```

### 场景三：实盘前安全检查

```
1. /code-review        → 检查所有代码
2. /security-review    → 安全问题排查
3. /verify             → 确认风控规则全部生效
4. /init               → 生成项目文档（给未来的自己看）
```

## 8. 常见问题

### Q: 我需要记住所有 Skill 吗？

**不需要。** 大部分时候你直接用自然语言说话，Claude 会自动判断该用哪个 Skill。斜杠命令只是快捷方式——你想用的时候可以用，不用也完全可以。

### Q: Skill 可以组合使用吗？

**可以。** 比如你可以说："帮我审查代码（code-review），然后跑起来（run）看看效果，最后确认改动生效了（verify）"——Claude 会按顺序执行。

### Q: 我什么时候该主动用 / 命令？

```
主动用斜杠命令的场景：
  /code-review   → 你刚写完一堆代码，心里没底
  /simplify      → 你感觉代码越来越乱
  /verify        → 你修了一个 bug，想确认修好了
  /deep-research → 你需要系统调研一个问题

不需要斜杠的场景：
  "帮我写一个函数"          → 直接说就行
  "这个变量是什么意思"      → 直接问
  "跑一下看看"             → 直接说（会自动用 /run）
```

### Q: 会不会用着用着就忘了？

**不会。** Claude 会在合适的时机主动提醒你。比如你写完代码后，它可能会说："这个改动涉及策略核心逻辑，要不要用 /code-review 审查一下？"
