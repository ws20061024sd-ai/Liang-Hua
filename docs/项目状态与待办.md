### 方案一：Bark（🏆 最推荐）

**原理**

```
run.py → HTTP请求 → Bark服务器 → iPhone通知
                        ↓
              Apple Push Notification Service (APNs)
                        ↓
                   你的iPhone弹出通知
```

**iPhone 端**：App Store 搜 "Bark" 安装，打开即得到一个推送 URL（如 `https://api.day.app/xxxx`）

**代码**：一行搞定

```python
import requests
requests.get(f"https://api.day.app/{YOUR_KEY}/量化信号/今日买入2只，卖出0只")
```

**效果**：

```
┌─────────────────────────┐
│ 📊 量化信号              │
│ 今日买入2只，卖出0只      │
│ 四川路桥 MA10上穿MA30    │
└─────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| 配置只需 1 分钟 | 仅支持 iPhone |
| 免费、无广告 | 不支持 Markdown 排版 |
| 电池耗电极低（走 APNs） | 消息长度有限制 |
| 开源，可自建服务器 | — |
| ⭐ 最推荐新手 | — |

---

### 方案二：钉钉机器人

**原理**

```
run.py → HTTP请求 → 钉钉服务器 → 钉钉群消息
```

**配置步骤**：

1. 建个钉钉群（只拉自己）
2. 群设置 → 智能群助手 → 添加自定义机器人
3. 复制 Webhook URL

**代码**：约10行

```python
import requests, json
webhook = "https://oapi.dingtalk.com/robot/send?access_token=xxx"
data = {
    "msgtype": "markdown",
    "markdown": {
        "title": "量化交易信号 2026-06-07",
        "text": "## 🟢 买入建议\n- 四川路桥(600039) MA10上穿MA30\n- 江苏银行(600919) MA10上穿MA30\n\n---\n## ⚠️ 持仓关注\n无"
    }
}
requests.post(webhook, json=data)
```

**效果**：钉钉群聊天消息，支持 Markdown 排版

| 优点 | 缺点 |
|------|------|
| 支持 Markdown（排版美观） | 需要安装钉钉 App |
| iPhone/Android 都支持 | 信号信息安全问题（自建群=只有自己看，无碍） |
| 消息长度无限制 | 需要建群、加机器人（多几步） |
| 可发多条，可 @自己 | 推送走钉钉通道，可能有延迟 |

## 📋 待实施：每日市场分析日报

### 需求

每天收盘后推送到钉钉，从宏观到微观三层分析：

```
宏观（指数）  →  四大指数 + 涨跌家数 + 市场状态
中观（板块）  →  行业/板块涨跌排名 + 资金流向
微观（个股）  →  信号股复盘 + 异动股 + 持仓关注
```

### 方案对比

#### 方案A：独立脚本 + 独立定时（推荐）

```
目录结构：
  analysis/
  ├── __init__.py
  ├── macro.py       # 宏观分析
  ├── sector.py      # 板块分析
  └── stock.py       # 个股分析

执行方式：
  1. python analysis.py  （独立脚本）
  2. crontab: 10 15 * * 1-5  → 每天 15:10（收盘后5分钟）

钉钉推送：
  独立一条消息（和交易信号分开），标题"📈 市场日报 2026-06-09"
```

| 优点 | 缺点 |
|------|------|
| 和交易信号解耦，独立迭代 | 多一个脚本和 cron |
| 分析逻辑复杂，独立文件好维护 | 拉数据需要额外网络请求 |
| 可分开发送（不想看分析时不打扰） | 运行时间稍长（10-30秒） |

#### 方案B：集成到 run.py

```
直接在 run.py 里加分析模块，信号+分析一条消息推送
```

| 优点 | 缺点 |
|------|------|
| 一个脚本搞定 | 消息太长，阅读体验差 |
| 不需要额外 cron | 代码混在一起难维护 |
| 数据复用（已拉过） | 分析出bug会影响信号推送 |

#### 方案C：分两条钉钉消息，但共用 run.py

```
run.py 执行完后 → 先发信号消息 → 再发分析消息
```

| 优点 | 缺点 |
|------|------|
| 一次运行，数据复用 | run.py 变重（多5-10秒） |
| 两条独立消息，阅读清晰 | 代码耦合 |
| 仅需一个 cron | — |

### 推荐：**方案A（独立脚本）**

理由：信号和分析是两类完全不同的信息——
- 信号是"要不要行动"（精简短）
- 分析是"理解市场"（详实长）
- 混在一起两头不讨好

### 分析内容设计

#### 宏观分析（数据源：AKShare指数接口）

```
📊 主要指数
  ├── 沪深300 / 上证 / 深证 / 创业板
  ├── 涨跌幅 + 成交量对比
  └── 指数MA状态（MA20/MA60）

📈 市场广度（成分股数据）
  ├── 涨跌家数（沪深300）
  ├── 涨幅中位数
  └── 总成交额

🔄 市场状态
  └── 复用防线一的 regime 判断
```

#### 板块分析（数据源：成分股按代码前缀+行业分群）

```
🏭 板块涨跌
  ├── 四大板块对比（主板/创业板/科创板）
  ├── 按股价分层（<10 / 10-30 / 30-50 / 50-100 / >100）
  └── 按成交额分层

💡 风格判断
  ├── 价值 vs 成长
  ├── 大盘 vs 小盘
  └── 防御 vs 进攻
```

#### 个股分析（数据源：本地 SQLite）

```
🎯 昨日信号复盘
  ├── 前一日信号股的实际涨跌
  ├── 信号准确度（涨了/跌了）
  └── 是否触发止损

🔥 异动个股
  ├── 涨幅前5 + 跌幅前5
  ├── 成交额异常放大
  └── 连续涨跌

📋 当前持仓关注（如果已开始跟单）
  ├── 浮盈/浮亏
  └── 是否接近止损/止盈
```
