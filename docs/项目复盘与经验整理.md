## 二、遇到的问题与解决方案（按发生顺序）

### 问题1：AKShare 东方财富 API 全部连不上

**现象**：首次 `run.py --init` 下载 300 只股票，全部失败。报 `ProxyError`。

**排查过程**：
```
1. 怀疑代理 → 检查系统代理（Clash on 127.0.0.1:7897）
2. 关代理 → 仍然失败，报 SSL 证书错误
3. 发现 Python 3.14 缺少根证书 → 运行 Install Certificates.command 修复
4. SSL 修复后 → 还是连不上，报 RemoteDisconnected
5. curl 测试 → 能用但 Python 不行 → 怀疑 TLS 或 IP 问题
6. 换数据源 → 新浪 source 能用 ✅
```

**根因**：多层叠加——代理拦截 + SSL 证书缺失 + 东方财富 API 不稳定。

**最终方案**：优先用新浪数据源（`stock_zh_a_daily`），东方财富作备用。新浪稳定且不需要额外配置。

**教训**：
- 数据源必须有备份，不能依赖单一 API
- 启动时清除代理环境变量
- 在 downloader.py 开头加 `os.environ` 代理清理

### 问题2：增量下载时涨跌幅全为 NULL

**现象**：下载单日数据后，所有股票的 pct_change 字段为 None。日报显示 0涨0跌，平均涨跌 NaN。

**根因**：
```
新浪数据源不提供涨跌幅字段，代码用 groupby().pct_change() 计算。
但增量下载只有 1 天新数据时，第一天没有前一天参照，pct_change() 返回 NaN。
如果下载 3 天数据 → 第一天 NaN，后两天正常。
如果只下载 1 天数据 → 全部 NaN。
```

**修复过程**：
```
尝试1：在 downloader 里用 fillna 补 → 单日数据时全 NaN，fillna 无效
尝试2：存完数据后用 SQL 从 DB 前一天收盘价反算 → ✅ 可行
实现：fix_pct_change() 函数，每次 download_all() 后自动执行
```

**教训**：
- 数据清洗必须在「数据完整」的基础上做，单行数据无法自计算
- 增量更新的边界条件（只有 1 天新数据）必须测试
- SQL 反算比 Python 计算更可靠（有完整的跨天数据）

### 问题3：日报指数与广度数据日期不匹配

**现象**：日报标题显示 6/9，广度数据显示 6/9，但指数数据显示的是 6/8。同一份报告中数据来自两天。

**根因**：
```
指数数据：从 AKShare API 实时拉取（可能有最新数据，也可能没有）
广度数据：从本地 SQLite 读取（固定是 DB 最新日期）

当 AKShare 的指数数据和 DB 的股票数据不是同一天时 → 错配
```

**修复过程**：
```
尝试1：让日报标注数据日期 → 只是标注，数据还是错的
尝试2：让指数 API 按 DB 日期查 → 找不到时回退最新
尝试3（最终）：所有指数统一用「最新日期中的最老值」
  - 比如 沪深300 有 6/9，深证成指 只有 6/8 → 全部用 6/8
  - 宁可全部用旧一天，也不能混用两天
```

**教训**：
- 多数据源的报告，必须统一数据时间点
- 标注错误不如防止错误

### 问题4：相同指数数据日期不一致

**现象**：6/9 日报中，深证成指用的是 6/8 数据，但创业板指用的是 6/9 数据。同一张表里指数来自两天。

**根因**：不同指数在 AKShare 的更新速度不同。深证成指可能 17:30 还没更新，创业板已经更新了。

**最终方案**（和问题3合并解决）：
```python
# 取所有指数最新日期的交集（最小值）
unified_date = min(df['date'].max() for df in index_data)
# 所有指数统一使用这个日期
```

### 问题5：部分股票下载失败，数据覆盖率不足

**现象**：300 只股票中只有 280 只有数据，20 只缺失。

**排查**：这 20 只不是停牌（同花顺可查到行情），是下载时被跳过了。

**根因**：下载器虽有 3 次重试，但遇到持续网络抖动时仍会遗漏。300 只 × 0.15s = 45 秒的下载过程中，部分请求可能恰好失败。

**修复**：加第二轮补漏——第一轮失败的股票在全部下载完后再集中重试（间隔 1 秒），成功率显著提高。

**教训**：
- 重试机制应该分两轮：快速重试 + 长间隔重试
- 下载完成后必须验证覆盖率（verify_data_quality）

---

### 问题6：服务器 SSH 连不上

**现象**：Claude Code 无法 SSH 到服务器，即使代理已关闭。

**根因**：macOS 系统级代理（Clash）配置在系统网络设置中，SSH 走系统代理，代理连不上目标。

**教训**：
- 开发环境应独立于系统的网络配置
- 远程操作命令应给用户自行执行，而非依赖 SSH 自动化

---

### 问题7：cron 日志目录不存在

**现象**：cron 配置了 `>> logs/cron.log`，但 `logs/` 目录未创建，cron 执行失败无输出。

**修复**：在服务器部署时 `mkdir -p logs/`。

**教训**：
- 部署 checklist 必须包含所有目录创建
- 写一个 `setup.sh` 自动建目录（待做）

### 问题8：策略信号过多（噪声）

**现象**：3 个策略并行后，卖出信号 135 条，推送消息过长。

**根因**：卖出信号包含大量「建议卖出但实际未持有的股票」，对用户没有意义。

**修复**：
```
短期：聚合 + 限制显示前 5 条
长期：卖出信号仅对「已持仓股票」展示（需要持仓跟踪系统）
```

## 五、代码模式总结

### 5.1 数据修复模式

```python
# 下载 → 修复 → 验证 的标准流程
download_all()        # 拉数据
fix_pct_change()     # 修数据
verify_data_quality() # 查数据
```

### 5.2 双重保障模式

```python
# run.py 和 report.py 各自独立执行数据检查
# 一个挂不影响另一个
run.py:
    fix_pct_change()
    verify_data_quality()

report.py:
    fix_pct_change()      # 再次修复（兜底）
    verify_data_quality() # 再次检查（兜底）
```

### 5.3 API 降级模式

```python
# 永远有备用方案
def download_stock_history():
    # 方案1：新浪（稳定，主用）
    df = ak.stock_zh_a_daily(...)
    if df is None:
        # 方案2：东方财富（备用）
        df = ak.stock_zh_a_hist(...)
    return df
```

### 5.4 静默 + 告警模式

```python
# 数据异常时不显示假数据，而是明确告警
if pct_change 全 NULL:
    return {'data_error': '涨跌幅数据全部缺失'}  # 不返回 0涨0跌
```

## 七、如果重来会怎么设计

1. **数据管道应该先完成**：先确保 300 只股票的完整历史数据在本地，再开始写策略。我们边写策略边修数据问题，多花了很多时间。

2. **指数数据应该进 DB**：4 个指数的日线应该和股票数据一起下载，彻底消除日期不一致。

3. **先单策略稳定再扩展**：双均线跑通后应该观察几天，再加速动量突破和均值回归。

4. **部署脚本化**：服务器的 Python 安装、目录创建、cron 配置应该一个 `setup.sh` 搞定，而不是手动逐步操作。

### 问题9：Mac crontab 残留导致双重推送

**现象**：21:00 收到正确推送，23:00 又收到一次旧数据的推送。

**根因**：部署服务器后未清理 Mac 本地 `crontab -l` 中的旧定时任务。两个机器同时跑。

**修复**：`crontab -r` 清理 + 推送末尾加 `📍 主机名` 标识来源。

**教训**：部署审查必须包括「旧环境清理」，不能只检查新环境。

### 问题10：深证成指/创业板指数据日期不一致

**现象**：同一份日报中，深证成指显示 6/8 数据，创业板指显示 6/9 数据。

**根因**：不同指数在 AKShare 的更新速度不同。各自取「最新」导致日期不统一。

**修复**：`unified_date = min(各指数最新日期)`——统一取最老日期，保证一致性。
