## 二、遇到的问题与解决方案（按发生顺序）

### 问题1：AKShare 东方财富 API 全部连不上

**现象**：首次 `run.py --init` 下载 300 只股票，全部失败。报 `ProxyError`。

**排查过程**：
```
1. 怀疑代理 → 检查系统代理（Clash on 127.0.0.1:7897）
2. 关代理 → 仍然失败，报 SSL 证书错误
3. 发现 Python 3.14 缺少根证书 → 运行 Install Certificates.command 修复
4. SSL 修复后 → 还是连不上，报 RemoteDisconnected
5. curl 测试 → 能用但 Python 不行 → 怀疑 TLS 或 IP 问题
6. 换数据源 → 新浪 source 能用 ✅
```

**根因**：多层叠加——代理拦截 + SSL 证书缺失 + 东方财富 API 不稳定。

**最终方案**：优先用新浪数据源（`stock_zh_a_daily`），东方财富作备用。新浪稳定且不需要额外配置。

**教训**：
- 数据源必须有备份，不能依赖单一 API
- 启动时清除代理环境变量
- 在 downloader.py 开头加 `os.environ` 代理清理

### 问题2：增量下载时涨跌幅全为 NULL

**现象**：下载单日数据后，所有股票的 pct_change 字段为 None。日报显示 0涨0跌，平均涨跌 NaN。

**根因**：
```
新浪数据源不提供涨跌幅字段，代码用 groupby().pct_change() 计算。
但增量下载只有 1 天新数据时，第一天没有前一天参照，pct_change() 返回 NaN。
如果下载 3 天数据 → 第一天 NaN，后两天正常。
如果只下载 1 天数据 → 全部 NaN。
```

**修复过程**：
```
尝试1：在 downloader 里用 fillna 补 → 单日数据时全 NaN，fillna 无效
尝试2：存完数据后用 SQL 从 DB 前一天收盘价反算 → ✅ 可行
实现：fix_pct_change() 函数，每次 download_all() 后自动执行
```

**教训**：
- 数据清洗必须在「数据完整」的基础上做，单行数据无法自计算
- 增量更新的边界条件（只有 1 天新数据）必须测试
- SQL 反算比 Python 计算更可靠（有完整的跨天数据）

---

### 问题3：日报指数与广度数据日期不匹配

**现象**：日报标题显示 6/9，广度数据显示 6/9，但指数数据显示的是 6/8。同一份报告中数据来自两天。

**根因**：
```
指数数据：从 AKShare API 实时拉取（可能有最新数据，也可能没有）
广度数据：从本地 SQLite 读取（固定是 DB 最新日期）

当 AKShare 的指数数据和 DB 的股票数据不是同一天时 → 错配
```

**修复过程**：
```
尝试1：让日报标注数据日期 → 只是标注，数据还是错的
尝试2：让指数 API 按 DB 日期查 → 找不到时回退最新
尝试3（最终）：所有指数统一用「最新日期中的最老值」
  - 比如 沪深300 有 6/9，深证成指 只有 6/8 → 全部用 6/8
  - 宁可全部用旧一天，也不能混用两天
```

**教训**：
- 多数据源的报告，必须统一数据时间点
- 标注错误不如防止错误

### 问题4：相同指数数据日期不一致

**现象**：6/9 日报中，深证成指用的是 6/8 数据，但创业板指用的是 6/9 数据。同一张表里指数来自两天。

**根因**：不同指数在 AKShare 的更新速度不同。深证成指可能 17:30 还没更新，创业板已经更新了。

**最终方案**（和问题3合并解决）：
```python
# 取所有指数最新日期的交集（最小值）
unified_date = min(df['date'].max() for df in index_data)
# 所有指数统一使用这个日期
```

### 问题5：部分股票下载失败，数据覆盖率不足

**现象**：300 只股票中只有 280 只有数据，20 只缺失。

**排查**：这 20 只不是停牌（同花顺可查到行情），是下载时被跳过了。

**根因**：下载器虽有 3 次重试，但遇到持续网络抖动时仍会遗漏。300 只 × 0.15s = 45 秒的下载过程中，部分请求可能恰好失败。

**修复**：加第二轮补漏——第一轮失败的股票在全部下载完后再集中重试（间隔 1 秒），成功率显著提高。

**教训**：
- 重试机制应该分两轮：快速重试 + 长间隔重试
- 下载完成后必须验证覆盖率（verify_data_quality）

### 问题6：服务器 SSH 连不上

**现象**：Claude Code 无法 SSH 到服务器，即使代理已关闭。

**根因**：macOS 系统级代理（Clash）配置在系统网络设置中，SSH 走系统代理，代理连不上目标。

**教训**：
- 开发环境应独立于系统的网络配置
- 远程操作命令应给用户自行执行，而非依赖 SSH 自动化

**修复**：在服务器部署时 `mkdir -p logs/`。

**教训**：
- 部署 checklist 必须包含所有目录创建
- 写一个 `setup.sh` 自动建目录（待做）

### 问题8：策略信号过多（噪声）

**现象**：3 个策略并行后，卖出信号 135 条，推送消息过长。

**根因**：卖出信号包含大量「建议卖出但实际未持有的股票」，对用户没有意义。

## 五、代码模式总结

### 5.1 数据修复模式

```python
# 下载 → 修复 → 验证 的标准流程
download_all()        # 拉数据
fix_pct_change()     # 修数据
verify_data_quality() # 查数据
```

### 5.2 双重保障模式

```python
# run.py 和 report.py 各自独立执行数据检查
# 一个挂不影响另一个
run.py:
    fix_pct_change()
    verify_data_quality()

report.py:
    fix_pct_change()      # 再次修复（兜底）
    verify_data_quality() # 再次检查（兜底）
```

### 5.3 API 降级模式

```python
# 永远有备用方案
def download_stock_history():
    # 方案1：新浪（稳定，主用）
    df = ak.stock_zh_a_daily(...)
    if df is None:
        # 方案2：东方财富（备用）
        df = ak.stock_zh_a_hist(...)
    return df
```

### 5.4 静默 + 告警模式

```python
# 数据异常时不显示假数据，而是明确告警
if pct_change 全 NULL:
    return {'data_error': '涨跌幅数据全部缺失'}  # 不返回 0涨0跌
```

## 七、如果重来会怎么设计

1. **数据管道应该先完成**：先确保 300 只股票的完整历史数据在本地，再开始写策略。我们边写策略边修数据问题，多花了很多时间。

2. **指数数据应该进 DB**：4 个指数的日线应该和股票数据一起下载，彻底消除日期不一致。

3. **先单策略稳定再扩展**：双均线跑通后应该观察几天，再加速动量突破和均值回归。

4. **部署脚本化**：服务器的 Python 安装、目录创建、cron 配置应该一个 `setup.sh` 搞定，而不是手动逐步操作。

5. **回测引擎优先于策略**：应该在写策略之前就把回测引擎搭好（含蒙特卡洛），这样每个策略上线前都能立刻跑完整验证，而不是事后补课。

6. **参数集中化从第一天开始**：settings.py 是项目第一天就创建的，但直到 v0.2.0 审查才发现大量参数散落在外。应该从一开始就强制「所有可调数字必须进 settings」。

7. **文档目录从一开始就分类**：15 个文件平铺在 docs/ 根目录，找了 7 天才重构。第一天就应该建子目录。

### 问题9：Mac crontab 残留导致双重推送

**现象**：21:00 收到正确推送，23:00 又收到一次旧数据的推送。

**根因**：部署服务器后未清理 Mac 本地 `crontab -l` 中的旧定时任务。两个机器同时跑。

**修复**：`crontab -r` 清理 + 推送末尾加 `📍 主机名` 标识来源。

**教训**：部署审查必须包括「旧环境清理」，不能只检查新环境。

### 问题10：深证成指/创业板指数据日期不一致

**现象**：同一份日报中，深证成指显示 6/8 数据，创业板指显示 6/9 数据。

**根因**：不同指数在 AKShare 的更新速度不同。各自取「最新」导致日期不统一。

**修复**：`unified_date = min(各指数最新日期)`——统一取最老日期，保证一致性。

### 问题11：信号复盘永远返回空（隐藏 Bug）

**现象**：日报「昨日信号复盘」段从不显示数据。命中率统计功能完全失效。

**根因**：[analysis/stock.py:19-26](analysis/stock.py#L19-L26) 中 `conn.close()` 在第 19 行调用，但第 26 行仍将已关闭的连接传给 `_review_signals(conn)`。函数内部尝试查询 → 抛异常 → 被 `except Exception: return []` 静默吞掉。

**教训**：
- `try/except: return []` 是双刃剑——它防止了崩溃，但也埋藏了 bug
- SQLite 连接管理应遵循「谁打开谁关闭」原则，严禁跨作用域传已关闭的连接
- 写代码时要问自己：如果这段永远不执行，会发现吗？

### 问题12：广度全 NULL 时崩溃（隐藏 Bug）

**现象**：当 `pct_change` 全 NULL 时，广度分析整个消失。

**根因**：[analysis/macro.py:143](analysis/macro.py#L143) 中引用了不存在的变量 `max_date`（实际变量名是 `data_date`），触发 `NameError` → 被外层 `except: return {}` 吞掉。

**教训**：同上——`try/except` 掩盖的 bug 比 crash 更可怕。crash 至少你能发现。

### 问题13：参数散落各处

**现象**：止损比例、仓位上限、数据加载天数、质量检查阈值等关键参数硬编码在各个源文件中，调参需要在多处修改。

**修复**：所有参数集中到 `config/settings.py`：
- 均值回归参数（`BB_PERIOD`, `BB_STD_DEV`）
- 仓位档位表（`POSITION_TIERS`）——用 `(上限, 单票%, 止损%, 持仓数)` 元组数组
- 策略运行器参数（`STRATEGY_DATA_DAYS`, `SELL_SIGNAL_LIMIT`）
- 数据质量阈值（`MIN_STOCK_COUNT`, `MIN_SECTOR_COUNT`, `MAX_EXTREME_PCT`）

**教训**：
- 项目初期"先写死再抽离"是正常的，但到 v0.2 必须集中化
- 用元组数组替代 if/elif 链，新增档位加一行即可

### 问题14：回测结论被数据区间彻底颠覆

**现象**：3 年回测（2023-2026）显示「择时改善 +14pp」，7 年回测（2019-2026）显示「择时恶化 -5pp」。

**根因**：
```
3 年数据 = 只覆盖了熊市（49% 弱势日）
  → 择时在熊市中表现英勇（躲过大跌）
  → 结论：择时是最大 Alpha 来源

7 年数据 = 覆盖完整牛熊（2019-2020 大牛市）
  → 择时在牛市中踏空了（均线滞后）
  → 结论：择时保护了熊市但害了牛市，净效果略负
```

**教训**：
- **少于 5 年的回测结论不可靠**——可能只覆盖了半个周期
- 绝不能根据短期回测做决策——你只会看到你想看到的
- 2019 年的牛市是 2023-2026 熊市的对称面——两者都必须看

### 问题15：回测性能瓶颈

**现象**：原始回测 3 种方案各自从头算策略信号，3 年数据跑 20 分钟，7 年预计 50 分钟。

**根因**：每种方案（无择时/二元择时/权重匹配）独立运行策略计算，但 95% 的计算量（策略信号生成）完全相同——只是在不同的过滤规则下反复算同样的事。

**修复**：
```
引入「预计算 + 复用」架构：
  1. 所有策略信号只计算一次 → 存入 dict（128,570 条）
  2. 三种方案从同一信号池中按不同规则筛选
  3. 速度提升 ~40%（27 分钟跑完 7 年）
```

**关键代码模式**：`{(date, code, strategy_name, style): {action, strength, price}}`

**教训**：
- 回测优化的黄金法则：找到重复计算，抽出来只算一次
- 时间换空间在这里不值得——多花 100MB 内存缓存信号，省几十分钟

### 问题16：蒙特卡洛方法论迭代

**现象**：第一次实现的蒙特卡洛（交易顺序打乱）结果不变——因为乘法交换律，最终结果相同。第二次（逐笔 bootstrap）结果爆炸——个别极端交易被重复抽取导致结果失真。

**根因**：
```
尝试1：打乱交易顺序
  → 1.0 × A × B × C = 1.0 × C × B × A（乘法可交换）
  → 1000 次模拟全一样 → 无用

尝试2：有放回抽样逐笔交易
  → 盈利交易被重复抽到 → 指数爆炸
  → 亏损交易被漏掉 → 收益虚高
  → 中位数年化 +9183% → 不合理

尝试3（最终）：月度块 bootstrap
  → 交易按 20 笔一组切块（模拟月度）
  → 有放回抽块 → 保留块内相关性
  → 合理的结果：中位数 +113%，最差 5% -44%
```

**教训**：
- 蒙特卡洛不是简单地"随机打乱"——方法选择决定了结果是否有意义
- 块抽样 > 逐笔抽样——因为保留了真实市场的时间相关性
- 如果蒙特卡洛结果看起来太好了，那一定是方法有问题

### 问题17：聚宽 API 版本差异

**现象**：聚宽回测模板报 `NameError: name 'current_data' is not defined`。

**根因**：聚宽新版 API 中 `current_data` 不再是全局变量，需改用 `get_current_data()` 函数调用。

**教训**：
- 不同平台的 API 差异是客观存在的——即使同一平台不同版本也可能变
- 写跨平台代码时先跑一个最小 hello world 验证环境

### 问题18：策略匹配用字符串匹配太脆弱

**现象**：`market_timing.py` 中用 `'趋势' in strategy or '突破' in strategy` 判断策略类型——加新策略时如果命名不一致，权重调节静默失效。

**修复**：在 `BaseStrategy` 增加 `style` 类属性（`'trend'` | `'reversion'`），`market_timing.py` 用策略名→style 映射表替代字符串匹配。

**教训**：
- 用字符串做逻辑判断是技术债——迟早会因改名而 bug
- 类型的区分应该用显式的属性/enum，而不是靠命名约定
