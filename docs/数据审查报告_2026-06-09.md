## 一、数据质量红线

```
数据是量化的血液。以下红线不可破：

1. 每日推送的数据必须和「今日」匹配（不能推昨天的数据冒充今天）
2. 所有分析组件（指数/广度/板块/个股/行业）必须来自同一天
3. 关键字段（pct_change）永远不能为 NULL
4. 股票覆盖率 ≥ 280只（允许 ~20 只停牌）
5. 指数数据必须有（不能整个表消失）
6. 行业数据缺失时标明原因，不显示 0涨0跌
```

## 二、已知风险点与解决方案

### 🔴 风险1：pct_change 全部 NULL

**触发条件**：新浪数据源不提供涨跌幅，增量下载单日数据时 pct_change() 算不出

**后果**：日报广度显示 0涨0跌、NaN均值

**解决**：
```
第一道：download_all() 后用 fix_pct_change() SQL反算
第二道：report.py 独立再跑一次 fix_pct_change()（兜底）
第三道：analysis/macro.py _analyze_breadth() 检测全NULL时返回 data_error 标记
第四道：notifier 看到 data_error 显示 ⚠️ 而非 0涨0跌
```

**验证**：`SELECT COUNT(*) FROM daily_kline WHERE date='2026-06-09' AND pct_change IS NULL` 必须返回 0

---

### 🔴 风险2：数据覆盖率不足（部分股票缺失）

**触发条件**：网络波动导致下载失败，重试后仍有遗漏

**后果**：市场广度/板块分析缺少样本，统计偏差

**解决**：
```
第一道：download_all() 第一轮 3次重试（退避 0.5s/1.0s）
第二道：失败股票第二轮补漏（间隔 1.0s）
第三道：verify_data_quality() 检查当天股票数 ≥ 280
第四道：不足时在日报顶部 🚨告警
```

**验证**：每天 cron 日志检查「下载失败: X 只 → 第二轮后剩余 Y 只」，Y 必须为 0

---

### 🔴 风险3：指数数据与广度数据日期不匹配

**触发条件**：指数从 AKShare 实时拉取（最新日期），广度从 DB 读取（可能是昨天的）

**后果**：日报显示 "6/9" 的标题 + 6/8 的指数数据 → 用户困惑

**解决**：
```
日报统一数据日期：
  1. get_data_date() 查询 DB 最新日期
  2. 所有分析组件传入相同 data_date
  3. 指数 API 查指定日期 → 找不到时回退最新 → 标注实际日期
  4. 数据日期 ≠ 今日时在日报标题下方 ⚠️标注
```

**验证**：日报中所有日期标注必须一致

---

### 🟡 风险4：行业数据缺失

**触发条件**：同花顺 THS API 失败（网络问题）

**后果**：日报缺少行业板块分析

**解决**：
```
第一道：daily report.py 调用 THS API，成功 → 缓存到 sector_history 表
第二道：API 失败时用 industry.py 关键词分类（本地数据，不需要网络）
第三道：sector.analyze() 返回的 industries 为 None 时不展示行业段
```

**验证**：日报中有 "🏭 行业板块" 段

---

### 🟡 风险5：数据不是今天的（延迟）

**触发条件**：
- 当天是节假日但 cron 照常跑
- AKShare 数据在 17:30 还没更新
- 网络问题下载失败

**解决**：
```
cron 定时 → 17:30 收盘后（给数据商足够时间发布）
周末检测 → weekday >= 5 自动退出，不推送
工作日无数据 → 可能是节假日 → 继续推送但标注「数据未更新到今日」
```

---

### 🟢 风险6：同花顺 API 极值数据

**触发条件**：THS API 返回的涨跌幅存在极值（如 624% 的异常数据已在 6/9 测试中出现）

**后果**：行业统计失真

**解决**：
```
在 _ths_industry_analysis() 中增加极值过滤：
  - 涨跌幅绝对值 > 100% → 标记异常，不参与排名
  - 打印日志告警
```

## 三、自动保障体系（完整版）

```
每天 17:30 cron → run.py
  ├── download_all()         ← 全量下载 + 第二轮补漏
  ├── fix_pct_change()       ← SQL 反算 NULL 涨跌幅
  └── verify_data_quality()  ← 日期+数量+NULL 三项检查

每天 17:35 cron → analysis/report.py
  ├── fix_pct_change()       ← 兜底修复
  ├── verify_data_quality()  ← 兜底检查
  ├── save_today_sectors()   ← 缓存行业数据
  ├── macro.analyze(data_date)   ← 统一日期
  ├── sector.analyze()           ← 含行业涨跌
  ├── sector_stocks()            ← 板块内个股
  ├── stock.analyze()            ← 个股异动
  └── 格式化 → 质量有问题时 🚨标注 → 推钉钉
```

## 五、健康检查命令

每天 17:35 后在服务器上跑以下命令，确认数据正常：

```bash
cd /root/Liang-Hua && ./venv/bin/python -c "
import sqlite3
from datetime import datetime
conn = sqlite3.connect('data/stocks.db')
today = datetime.now().strftime('%Y-%m-%d')

# 检查1：最新日期
max_date = conn.execute('SELECT MAX(date) FROM daily_kline').fetchone()[0]
print(f'1. 最新数据: {max_date} {\"✅\" if max_date == today else \"❌ 非今日\"}')

# 检查2：股票数量
cnt = conn.execute('SELECT COUNT(DISTINCT code) FROM daily_kline WHERE date=?', (max_date,)).fetchone()[0]
print(f'2. 股票数量: {cnt}/300 {\"✅\" if cnt >= 280 else \"❌ 不足\"}')

# 检查3：pct_change NULL
nulls = conn.execute('SELECT COUNT(*) FROM daily_kline WHERE date=? AND pct_change IS NULL', (max_date,)).fetchone()[0]
print(f'3. 涨跌幅NULL: {nulls} {\"✅\" if nulls == 0 else \"❌ 有NULL\"}')

# 检查4：行业缓存
sc = conn.execute(\"SELECT COUNT(DISTINCT name) FROM sector_history WHERE date=?\", (max_date,)).fetchone()[0]
print(f'4. 行业数据: {sc}个 {\"✅\" if sc >= 80 else \"⚠️ \"+str(sc)+\"个\"}')

# 检查5：信号记录
sig = conn.execute('SELECT COUNT(*) FROM signal_history WHERE date=?', (max_date,)).fetchone()[0]
print(f'5. 信号记录: {sig}条 {\"✅\" if sig > 0 else \"⚠️ 无信号\"}')

conn.close()
"
```
