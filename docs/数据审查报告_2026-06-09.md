## 一、发现的问题

### 🔴 致命问题：pct_change 大面积 NULL

**现象**：

| 日期 | 本地 | 服务器 | pct_change |
|------|:--:|:--:|:--:|
| 6/9 | — | 280行 | ❌ 全NULL → 已修复 ✅ |
| 6/8 | 179行 | ? | ❌ 全NULL |
| 6/5 | 300行 | 300行 | ✅ 正常 |
| 6/4及以前 | 300行 | 300行 | ✅ 正常 |

**根因**：

新浪数据源（AKShare `stock_zh_a_daily`）**不提供涨跌幅字段**。我们的下载器用 `groupby().pct_change()` 自动计算，但增量下载时只有 1-2 天新数据，第一天的 `pct_change()` 永远是 NaN（没有前一天作参照）。

```
下载 6/5~6/8 的数据：
  6/5: 有前一天(6/4)在批次内 → 能算 ✅
  6/8: 前一天(6/5)也在批次内 → 能算 ✅

但如果只下载 6/8~6/8（增量1天）：
  6/8: 批次内没有前一天 → NaN ❌
```

**已修复**：`fix_pct_change()` 函数——存完数据后用 SQL 从数据库前一天收盘价补算。服务器已修复 880 条。

---

### 🔴 致命问题：日报显示 0涨/0跌

**现象**：

```
📈 市场广度
数据日期：2026-06-09
- 上涨: 0 | 下跌: 0    ← 全是0
- 平均涨跌: nan%       ← NaN
```

**根因**：`analysis/macro.py` 的 `_analyze_breadth()` 统计 `pct_change > 0` 和 `< 0` 的数量。当 `pct_change` 全是 NULL 时，统计结果全是 0，均值是 NaN。

**修复链路**：
1. `fix_pct_change()` 修复 pct_change ← 已完成
2. 日报在 pct_change 全 NULL 时显示「数据异常」而非 0 ← 待修

### 🟡 中等问题：数据覆盖率不足

**现象**：

| 日期 | 应有 | 实有 | 缺失 |
|------|:--:|:--:|:--:|
| 6/8 | 300 | 179 | 121只（40%） |
| 6/5 | 300 | 300 | 0 |
| 6/9（服务器） | 300 | 280 | 20只 |

6/8 缺失 40% 的数据。原因可能是：
- 6/8 是大跌日，部分股票停牌
- 下载时网络不稳定，部分失败后未重试

**现状**：下载器有 3 次重试（退避 0.5s/1.0s），但缺失 40% 说明重试不够。

---

### 🟡 中等问题：信号日期单一

**现象**：

```
信号日期范围: 2026-06-09 ~ 2026-06-09  ← 只有今天
```

信号历史表只有今天的 161 条记录（因为信号持久化今天才上线，之前的信号不存在数据库中）。

**影响**：日报的「昨日信号复盘」永远为空，因为查不到昨天的信号。

**解决**：这是正常的——信号持久化只上线了1天，需要时间积累。至少需要 3-5 个交易日才能看到复盘数据。

---

### 🟡 中等问题：信号数量过多（噪声）

**现象**：

```
6/9 信号: 88条 passed + 73条 blocked = 161条
  - 其中买入 12 条（全部被大盘择时屏蔽）
  - 卖出 85 条（聚合后11条）
```

3个策略 × 300只股票 × 每个策略多个卖出条件 = 大量卖出信号。虽然聚合后压到 11 条，但终端/日志还是很长。

**优化方向**：卖出信号只有在用户有持仓时才展示（需要持仓跟踪系统）。

---

### 🟢 小问题：大盘择时数据时效

**现象**：

```
沪深300: 4713.64 | MA20: 4880.14 | MA60: 4732.42
```

但数据库最新数据是 6/8 或 6/9，而指数 4713.64 看上去不像是 6/9 的收盘价。

**根因**：`market_timing.py` 的 `get_market_regime()` 先查本地 DB 的 000300 数据（没有），然后 fallback 到 AKShare 实时拉取（拉到的是最近一个交易日的指数数据，可能是 6/8）。

**解决**：大盘择时用的指数数据需要明确标注取自哪个日期。

## 三、数据质量保障机制（完整方案）

### 3.0 设计原则

```
数据是量化的血液。以下三条红线不可破：

1. 每日推送的数据必须和「今日」匹配（不能推昨天的）
2. 所有分析组件（指数/广度/板块/个股）必须来自同一天
3. 关键字段（pct_change）不能为 NULL
```

### 3.1 五层保障体系

```
第一层：下载时
  ├── 300只逐个下载，每只最多重试3次（退避 0.5s/1.0s）
  └── 请求间隔 0.15s，避免 API 限流

第二层：补漏轮
  ├── 第一轮失败的股票进入第二轮
  ├── 每只间隔 1.0s 重试
  └── 失败的打印告警

第三层：pct_change 修复
  ├── fix_pct_change() 用 SQL 从数据库前一天收盘价反算
  └── 每天 download_all() 后自动执行

第四层：质量检查 verify_data_quality()
  ├── 检查1: 最新数据日期 == 今天？（周末跳过）
  ├── 检查2: 当天股票数 >= 280只？
  ├── 检查3: pct_change 有 NULL？
  └── 任一不通过 → 打印告警

第五层：日报一致性
  ├── 日报生成前查询 DB 最新数据日期
  ├── 所有组件（指数/广度/板块/个股）统一使用该日期
  ├── 数据非今日时在推送中明确标注
  └── pct_change 全 NULL 时显示「数据异常」而非 0涨0跌
```

### 3.2 执行时序

```
17:30 cron 触发 run.py
  │
  ├── 1. download_all()        ← 第一层+第二层
  ├── 2. fix_pct_change()      ← 第三层
  ├── 3. verify_data_quality() ← 第四层
  ├── 4. 大盘择时
  ├── 5. 策略运行
  ├── 6. 信号汇总
  ├── 7. 信号持久化
  └── 8. 钉钉推送信号

17:35 cron 触发 analysis/report.py
  │
  ├── 1. get_data_date()       ← 查 DB 最新日期
  ├── 2. 如果 != 今天 → 标注
  ├── 3. macro.analyze(date)   ← 统一日期
  ├── 4. sector.analyze()       ← 用 DB 最新日期
  ├── 5. stock.analyze()        ← 用 DB 最新日期
  └── 6. 钉钉推送日报
```

### 3.3 防御代码清单

| 位置 | 防御内容 | 触发条件 |
|------|------|------|
| `downloader.py` download_all | 第二轮重试 | 有股票下载失败 |
| `downloader.py` fix_pct_change | SQL反算NULL涨跌幅 | 每次自动执行 |
| `downloader.py` verify_data_quality | 3项检查 | 每次自动执行 |
| `analysis/macro.py` _analyze_breadth | pct_change全NULL时标记error | NULL=100% |
| `analysis/sector.py` analyze | pct_change全NULL时返回error | NULL=100% |
| `analysis/report.py` get_data_date | 查DB实际日期 | 每次执行 |
| `notifier/dingtalk_report.py` | 数据异常时显示⚠️ | data_error存在 |

### 3.4 数据异常时的用户提示

```
正常情况的日报：
  ## 📈 量化市场日报 2026-06-09
  数据日期：2026-06-09
  上涨: 186 | 下跌: 88

数据延迟（cron跑了但数据未更新）：
  ## 📈 量化市场日报 2026-06-09
  > ⚠️ 数据日期: 2026-06-08（非今日 2026-06-09）

涨跌幅全部缺失：
  📈 市场广度
  > ⚠️ 涨跌幅数据全部缺失，无法统计广度
```
