## 一、本轮发现的核心问题

### 🔴 crontab 残留导致双重推送

**现象**：21:00 收到正确推送，23:00 又收到一次基于旧数据的推送。

**根因**：Mac 本地的 `crontab -l` 仍保留着旧的定时任务 `5 15 * * 1-5 python run.py`。服务器部署后没有清理本地 crontab。

**为什么之前审查没发现**：审查清单只覆盖代码和数据，不覆盖操作系统级的定时任务。

**修复**：`crontab -r` 清理本地定时。今后每台机器的 crontab 必须在部署文档中明确登记。

## 二、完整审查清单（七层）

### 第1层：运行环境审查

```
□ crontab -l       ← 检查每台机器的定时任务，确保只有一份
□ launchd list     ← macOS 可能有额外的定时作业
□ 进程列表          ← ps aux | grep python，确认没有残留的运行实例
□ 系统时区          ← 所有机器统一 UTC+8
```

**规则**：只有服务器 crontab 能自动运行 `run.py`。其他一切自动运行都是 bug。

### 第2层：代码审查

```
□ 每个函数是否有明确的输入输出？
□ 错误路径是否被处理？
□ 重试逻辑是否有效？
□ SQL 查询是否有索引？
□ 新增模块是否已注册？
□ 删除的代码是否还有残留引用？
```

### 第3层：数据质量审查

```
□ pct_change 不为 NULL            ← 自动：fix_pct_change
□ 股票覆盖率 ≥ 280                 ← 自动：verify_data_quality
□ 最新数据日期 = 今天（交易日）     ← 自动：verify_data_quality
□ 行业数据 ≥ 80 个                ← 自动：verify_data_quality
□ 涨跌幅极值过滤                   ← 自动：THS API >50% 排除
□ 指数与广度日期一致               ← 自动：unified_date
□ 日报日期标注正确                 ← 自动：get_data_date
```

**操作**：每次改代码后，跑一次 `python analysis/report.py`，观察输出中的 ✅/❌。

### 第4层：推送审查

```
□ 推送是否来自正确的机器？     ← 新增：📍 hostname 标识
□ 一天只推一次？               ← 部署完清理旧 crontab
□ 推送内容与终端输出一致？
□ 数据日期标注正确？
```

**操作**：每次部署后，等推送到达，对比终端输出。

### 第5层：部署审查

```
□ 服务器代码是否是 git 最新？   ← git log --oneline -3
□ 服务器依赖是否完整？          ← pip list
□ crontab 是否正确？           ← crontab -l
□ 日志目录是否存在？            ← ls logs/
□ 数据库路径是否可写？          ← touch data/stocks.db
□ 本地 crontab 已清理？         ← crontab -l 应为空
```

### 第6层：稳定性审查（新增）

```
□ 同一脚本跑两次，结果是否一致？（幂等性）
   测试：python run.py --no-update 两次，信号数量应相同
□ 数据库 WAL 模式是否正常？
□ 磁盘空间是否充足？
□ 日志文件是否在增长（未无限膨胀）？
```

### 第7层：策略调试时的审查（面向未来）

```
当我们开始频繁调整策略参数时：

□ 策略参数变更 → 必须本地测试 → 回测验证 → 再上服务器
□ 新策略上线 → 先在 --no-update 模式下跑 → 观察信号合理性 → 再启用 cron
□ 策略测试期间 → 测试信号和正式信号分开推送
□ 参数调整 → 在 git commit message 中注明改了什么参数、为什么改
□ 每次调参后 → 跑一次回测对比前后表现
```

## 三、防范双重推送的三道锁

```
锁1：部署后清理本地 crontab
     crontab -r  或  crontab -l | grep -v run.py | crontab -

锁2：推送带来源标识
     现在每条钉钉消息末尾有「📍 VM-0-17-opencloudos | 2026-06-09」
     如果来自 Mac → 立刻知道有问题

锁3：run.py 加主机名检查
     如果 HOSTNAME 不是服务器 → 标记 「⚠️ 非生产环境」但不阻止
     （防止调试时完全收不到推送）
```

## 五、策略迭代规范（面向未来）

当我们进入策略调试阶段，最怕的就是「调参 → 出 bug → 用户发现 → 回头修」。

### 本地测试流程

```
1. 改策略参数 → 在本地跑 run.py --no-update
   → 观察信号数量是否合理（不是 0 也不是 500）

2. 跑回测 → backtest/simple_backtest.py
   → 对比新旧参数的回测指标

3. 拉取服务器最新数据 → git pull + 同步 DB
   → 用真实最新数据分析信号质量

4. 确认无误 → git commit + push → 服务器 git pull
```

### 参数管理

```
单一策略的参数集中在一处：
  config/settings.py  ← 所有策略参数唯一来源

变更记录在 git commit message 中：
  "param: MA双均线 10/30 → 5/20，回测夏普+0.2"

禁止：
  ❌ 在策略代码里硬编码参数
  ❌ 不跑回测直接改参数上线
  ❌ 同时改多个策略的参数
```

### 出问题时的回退

```
git log --oneline -5          # 找到上一个稳定版本
git revert <问题commit>       # 回退
git push                      # 推送回退
服务器 git pull               # 立即生效
```
