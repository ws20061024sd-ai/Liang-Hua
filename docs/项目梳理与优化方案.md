## 一、🔴 已发现 Bug（必须修复）

### Bug 1：信号复盘永远返回空（stock.py L19-26）

**位置**：[analysis/stock.py:19-26](analysis/stock.py#L19-L26)

**问题**：`conn.close()` 在第 19 行关闭连接，第 26 行又将已关闭的 `conn` 传给 `_review_signals(conn)`。`_review_signals` 内部尝试查询 SQLite → 抛异常 → 被 `except Exception: return []` 静默吞掉。

```python
# stock.py L12-26 当前代码
conn = sqlite3.connect(settings.DB_PATH)
df = pd.read_sql_query(...)
conn.close()                           # ← 第19行：已关闭

if df.empty:
    return {}

top_gainers = _top_movers(df, n=5, ascending=False)
top_losers = _top_movers(df, n=5, ascending=True)
signal_review = _review_signals(conn)  # ← 第26行：使用已关闭的连接！
```

**影响**：日报「🎯 昨日信号复盘」段永远不显示数据。信号命中率统计功能完全失效。

**修复**：`conn.close()` 移到 `_review_signals(conn)` 调用之后。

---

### Bug 2：广度全 NULL 时 crash 报 NameError（macro.py L143）

**位置**：[analysis/macro.py:143](analysis/macro.py#L143)

**问题**：`_analyze_breadth()` 在 `pct_change` 全 NULL 分支中引用了不存在的变量 `max_date`（实际变量名是 `data_date`）。

```python
# macro.py L136-144
if null_count == total:
    return {
        ...
        'data_date': str(max_date),  # ← NameError: max_date 未定义
        'data_error': '涨跌幅数据全部缺失，无法统计广度',
    }
```

**影响**：当涨跌幅全 NULL 时，函数抛 `NameError` → 被外层 `except: return {}` 吞掉 → 广度分析段整个消失。且即使修复变量名，`data_date` 此时是 `pd.Timestamp` 不是 `str`。

**修复**：`str(max_date)` → `str(data_date)`。

---

### Bug 3：均值回归策略参数硬编码（mean_reversion.py L32）

**位置**：[strategies/mean_reversion.py:32](strategies/mean_reversion.py#L32)

**问题**：`MeanReversionStrategy.__init__` 的 `period=20, std_dev=2.0` 直接硬编码，不从 `settings.py` 读取。而 `MaCrossStrategy` 和 `MomentumBreakoutStrategy` 都正确地从 settings 读取参数。

```python
# mean_reversion.py L32 — 硬编码
def __init__(self, period: int = 20, std_dev: float = 2.0):

# 对比 ma_cross.py L27 — 正确做法
def __init__(self, fast_period: int = None, slow_period: int = None):
    self.fast_period = fast_period or settings.MA_FAST
    self.slow_period = slow_period or settings.MA_SLOW
```

**影响**：违反「参数必须集中在 settings.py」规则。调参时需要在两个地方改，容易遗漏。

**修复**：在 `settings.py` 增加 `BB_PERIOD=20, BB_STD_DEV=2.0`，策略从 settings 读取。

### 缺陷 4：signal_store 日期逻辑错误

**位置**：[run.py:259-260](run.py#L259-L260)

## 三、🟢 代码质量问题

### 问题 1：全系统使用 print() 而非 logging

所有 19 个 .py 文件都用 `print()` 输出。生产环境中：
- cron 日志无时间戳，难以排查问题时间线
- 无法按级别过滤（DEBUG/INFO/WARNING/ERROR）
- 无法同时输出到文件和终端

**建议**：引入 `logging` 模块，`run.py` 和 `report.py` 配置双 handler（文件 + 控制台）。

---

### 问题 2：批量加载 200 天数据的性能瓶颈

[engine/runner.py:61](engine/runner.py#L61) 每次为 300 只股票各加载 200 天数据（约 60,000 行），但策略只需要：
- 双均线：最近 30 天
- 动量突破：最近 21 天
- 均值回归：最近 61 天

**建议**：`days` 参数改为 `max(p.slow_period for p in strategies) + 10` 或统一设为 70 天，减少数据传输量 ~65%。

---

**位置**：[analysis/sector_trend.py:87-88](analysis/sector_trend.py#L87-L88)

df = pd.read_sql_query("""

## 四、🟡 功能缺失

### 缺失 1：run.py 崩溃无告警

如果 `run.py` 在 21:00 因任何异常崩溃（网络断、AKShare API 变更、DB 损坏），用户不会收到任何通知——cron 静默失败。需要加一个独立的健康心跳检测：如果到了 21:10 还没收到信号推送，自动发一条告警。

**建议方案**：服务器上增加第三条 cron（21:10），检查今天 signal_history 是否有新记录，没有则发告警推送。

---

### 缺失 2：数据库无自动备份

每天写入数据，但没有任何备份。SQLite 文件被损坏则全丢。

**建议**：每日 cron 之后自动 cp 到 `data/backups/stocks_$(date +%Y%m%d).db`，保留最近 7 天。

---

### 缺失 3：回测只测了双均线

[backtest/simple_backtest.py](backtest/simple_backtest.py) 标题写"三策略六方案对比"，但代码只测试了 `MaCrossStrategy`。动量突破和均值回归的回测数据是手动推算的（在回测报告中标注了 `*`）。

---

### 缺失 4：无交易日历

周末检测用的 `datetime.now().weekday() >= 5`，但法定节假日（春节、国庆等）无法检测，会导致节假日推送旧数据。

## 五、🟢 策略逻辑审视

### 双均线（ma_cross.py）✅

逻辑正确。strength 用均线分离度计算合理。有停牌跳过和 NaN 保护。

### 动量突破（momentum_breakout.py）⚠️

- 买入用 20 日最高价 + 2% buffer，合理
- 卖出用 `exit_period` 日最低价（默认 20 日），与买入对称
- **问题**：`exit_period` 参数在 settings 中与 `lookback` 共用同一个值（`MOMENTUM_EXIT_PERIOD = 20`），但这是合理的默认行为
- **问题**：卖出 strength 硬编码为 0.5，不同于买入的动态计算

### 均值回归（mean_reversion.py）⚠️

- 买入条件：`close < bb_lower AND ma60_rising` ✅ 有趋势过滤
- **问题**：L70 计算了 `prev_below_lower` 但 L71 的 `signal_buy` 没有使用它——意味着只要当前 `below_lower` 就持续发信号，而非仅在首次下穿时触发。这可能导致同一股票连续多天发买入信号。这是设计选择（连续超卖可能持续发信号），但会产生重复信号噪声。
