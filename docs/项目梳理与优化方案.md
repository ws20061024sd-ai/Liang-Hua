## 二、已发现的问题

### 🔴 严重问题

#### 1. 策略运行器性能瓶颈

**位置**：[runner.py:62](engine/runner.py)

```python
for i, (_, stock) in enumerate(stocks.iterrows()):
    df = get_stock_data(code, days=200)  # 每只股票都打开/关闭一次 SQLite 连接
```

`get_stock_data()` 内部每次调 `sqlite3.connect()` + query + close。300只股票 = 300次数据库连接/断开。实测占运行时间的 60-70%。

**修复**：在 runner 层面打开一个连接，传入策略循环复用。

```python
conn = sqlite3.connect(settings.DB_PATH)
for stock in stocks:
    df = get_stock_data_from_conn(conn, code, days=200)
conn.close()
```

**影响**：每次运行多花 2-5 秒。

---

#### 2. 信号未持久化——无法复盘

**位置**：[downloader.py:42](data_fetcher/downloader.py) / [runner.py](engine/runner.py)

数据库里有 `signal_history` 表，但**没有任何代码往里面写数据**。信号生成了、推送了、然后丢了。

后果：
- 无法查看历史信号
- 无法统计信号准确度（涨了几个、跌了几个）
- 「昨日信号复盘」永远是空的

**修复**：在 `run.py` 生成信号后，写入 `signal_history` 表。之后 `analysis/stock.py` 就可以做真正的信号复盘。

---

#### 3. 分析日报依赖实时数据——收盘后立即跑可能拉不到

**位置**：[analysis/macro.py](analysis/macro.py)

日报 15:10 运行，但 AKShare 的日线数据可能要到 15:15-15:30 才更新。概率性拿到的是前一天数据而非当天收盘数据。

**修复**：
- 日报的「指数数据」用本地 SQLite（已经通过 15:05 cron 更新）
- 或者在 `_fetch_indices()` 中检查数据日期，如果是旧的则标注

#### 5. 数据源降级不可见

**位置**：[downloader.py:153-179](data_fetcher/downloader.py)

新浪源失败时静默切换到东财，但如果两个都失败，用户看不到任何日志（除非打开 verbose）。批量下载时，失败信息只打印前3条。

**修复**：增加最终的失败列表摘要（已有雏形，加上失败股票列表即可）。

---

#### 6. 涨跌幅计算有 NaN 风险

**位置**：[downloader.py:194-195](data_fetcher/downloader.py)

```python
if df['pct_change'].isna().all():  # 只有全部为 NaN 才计算
    df['pct_change'] = df.groupby('code')['close'].pct_change() * 100
```

如果新浪源提供了部分 `pct_change` 但有少数 NaN，`all()` 返回 False，NaN 原样保留。分析模块读取 NaN 会导致异常。

**修复**：改为 `any()` 或直接用 `fillna(0)`。

---

#### 7. 无交易日推送——用户可能疑惑

**位置**：[run.py:205](run.py)

当策略没产生任何信号时（偶尔发生），钉钉不推送。用户 15:05 过后没收到消息会疑惑「是挂了还是没信号？」

**修复**：即使无信号也推送一条简短消息：「今日无交易信号」。

---

#### 8. 重复的 tier_label 判断

**位置**：[run.py:212](run.py) 和 [notifier/dingtalk.py:65](notifier/dingtalk.py)

资金档位判断逻辑在两处重复。新增档位需要改两处。

**修复**：统一到 `config/settings.py` 的工具函数中。

## 三、性能分析

### 当前耗时拆解（300只股票）

| 阶段 | 耗时 | 瓶颈 |
|------|:---:|------|
| 成分股列表下载 | ~0.2s | 网络 |
| 数据增量更新 | ~30s | 300次 HTTP 请求 |
| 策略运行 | ~5s | 300次 SQLite 连接（问题1） |
| 风控过滤 | ~0.1s | — |
| 钉钉推送 | ~0.3s | 网络 |
| **合计** | **~35s** | — |

> 数据更新在服务器上增量模式通常为 0s（已是最新），实际每日运行 5-8 秒。

### 优化空间

| 优化项 | 预计节省 | 复杂度 |
|--------|:------:|:-----:|
| SQLite 连接复用（问题1） | 3-4 秒 | 低 |
| 指数数据本地缓存 | 1-2 秒 | 低 |
| 成分股列表缓存 | 0.2 秒 | 低 |
| 并发下载（asyncio） | 15-20 秒 | 高（暂不需要） |

> **结论**：目前性能完全够用（8 秒 vs 60 秒可接受范围），不必过早优化。修问题 1 即可。

## 四、可靠性分析

### 当前保护机制

| 场景 | 是否有保护 | 说明 |
|------|:--:|------|
| 网络断开 | ✅ | 重试3次 + 退避 |
| 数据源失效 | ✅ | 新浪→东财双源 |
| 数据库损坏 | ✅ | WAL 模式 |
| API 限流 | ⚠️ | 0.15s 间隔，但无明确检测 |
| 钉钉限流 | ⚠️ | 20条/分钟，当前远未触及 |
| 指数数据缺失 | ✅ | fallback 到 AKShare 实时 |
| 进程崩溃 | ❌ | 无守护或监控 |
| 磁盘满 | ❌ | 无检测 |
| 节假日误跑 | ❌ | 不会影响数据，会跑空 |

### 建议增强

1. **加一个健康检查定时任务**：每周一检查数据库行数是否增长，没增长就告警
2. **节假日日历**：接入 A 股交易日历，节假日跳过运行（可选，目前浪费不大）

## 五、架构评价

### 做得好的

1. **可插拔策略接口**：[base_strategy.py](strategies/base_strategy.py) 设计干净，加新策略只需继承 + 注册
2. **三道防线分层**：大盘择时 → 基础风控 → 仓位控制，顺序正确
3. **配置集中管理**：settings.py 一处改，全局生效
4. **数据源双活**：新浪主 + 东财备，提高了可用性
5. **信号 vs 分析分离**：run.py 负责行动信号，analysis/ 负责日报，职责清晰
6. **小资金适配**：按本金分档，考虑了现实约束

### 待改进

1. **runner.py 太重**：数据获取 + 策略执行 + 信号排序混在一个函数里
2. **全局状态缺失**：没有「当前持仓」、「已发出信号」、「资金曲线」等状态管理
3. **策略间无协作**：目前只有 1 个策略，但如果多策略，没有信号汇总/冲突处理（方案里写了但没实现）
4. **测试为零**：没有任何单元测试或集成测试
5. **没有交易日历**：周末和节假日照常跑（虽无害但浪费）
